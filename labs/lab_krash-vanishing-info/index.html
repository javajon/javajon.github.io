Vanishing Information

Challenge your Kubernetes skills and don't lose critical patient data.

This lab is part of a series that challenges your skills with running applications on Kubernetes. This particular lab focuses on one of the most critical aspects of Kubernetes storage: persistent volumes and data persistence.

In this lab, you are provided with a working Kubernetes cluster. You also have a telemedicine prescription refill service that appears to be running correctly. However, despite all components showing as Running, patient medication orders vanish every time pods restart. Your mission is to investigate why critical prescription data isn’t persisting and fix the underlying storage configurations.

In this lab, you will learn how to:

    ☐ Investigate storage-related issues in Kubernetes applications
    ☐ Use kubectl commands to diagnose persistent volume problems
    ☐ Understand the relationship between pods and data persistence
    ☐ Apply fixes to ensure data survives pod restarts
    ☐ Verify that your storage solution works correctly

Data persistence in Kubernetes can be deceptively complex. An application might appear fully functional until a pod restart reveals that all user data has vanished. This scenario mimics real production environments where storage misconfigurations can lead to data loss and compliance violations.

This lab features a realistic healthcare architecture:

The telemedicine prescription service is a modern healthcare application with a Python Flask portal for patients to manage prescription refills and a Node.js backend API that handles medication orders. The system is designed for easy use with click-only interfaces—no typing required.

When everything works correctly, patients can add medications to their refill cart and select insurance options and their orders persist across system maintenance. When storage configurations fail, critical prescription data vanishes with every pod restart.
